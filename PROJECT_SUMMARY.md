# Adaptive CRAG 项目总结与简历写法

## 1. 项目定位

**项目名称**：Adaptive CRAG：轻量化自适应检索增强生成系统

**一句话介绍**：

> 设计并实现一个面向长文档分析场景的轻量化 Corrective RAG 系统，支持文档解析、混合检索、证据评级、可选联网补偿、沙箱代码分析、报告生成与引用校验。

这个项目不是简单的“大模型问答 Demo”，而是一个有完整工程链路的 AI 应用系统。它覆盖了从文档进入系统，到生成可引用研究报告的完整流程。

---

## 2. 技术栈

- **语言与框架**：Python, Streamlit
- **流程编排**：LangGraph
- **检索系统**：ChromaDB, BM25, RRF
- **文档解析**：PyMuPDF
- **Embedding**：SentenceTransformers
- **LLM 接入**：OpenAI-compatible API, DashScope / 百炼等
- **安全执行**：AST import 白名单、沙箱执行、环境变量隔离
- **验证体系**：unittest, smoke test, benchmark runner

---

## 3. 系统架构与数据流

项目整体数据流：

```text
文档上传
  → 文档解析
  → 文本清洗
  → 切片 chunk
  → 建立索引
  → 混合检索
  → 证据评级
  → 可选联网补偿
  → 可选代码分析
  → 沙箱执行
  → 报告生成
  → 引用校验
```

核心模块关系：

```text
app/                  Streamlit 页面层
adaptive_crag/
  ingestion/          文档解析、PDF 切片、表格提取
  retrieval/          向量检索、BM25、RRF 融合、证据构建
  agents/             Router / Grader / Analyzer / Report 等 Agent
  graph/              LangGraph 状态机与节点流转
  sandbox/            代码安全检查与隔离执行
  evaluation/         benchmark、评分、测试集
  config/             配置管理与环境变量读取
```

这个结构的好处是：每个模块职责清晰，可以单独替换、单独测试，不会把 UI、检索、LLM prompt 和安全执行混在一起。

## 4. 已完成的核心功能

### 4.1 文档解析与切片

已实现：

- PDF / TXT / Markdown / DOCX 等文件解析
- PDF 页码级 chunk
- 文本清洗与页眉页脚过滤
- 小 chunk 合并
- chunk 来源信息补全
- PDF 表格提取并追加进 chunk 文本

这解决了两个真实问题：

1. LLM 只看到碎片，不知道片段来自哪篇文档、哪一页。
2. 论文表格中的定量数据无法进入检索链路。

### 4.2 混合检索

系统使用：

```text
向量检索 + BM25 关键词检索 + RRF 融合排序
```

这样做的价值：

- 向量检索适合语义相关内容。
- BM25 适合精确词、缩写、方法名、指标名。
- RRF 融合比单一路径更稳定。
- 关闭重型 reranker，避免下载大模型，保持项目轻量可运行。

### 4.3 CRAG 自适应流程

系统不是直接把检索结果交给 LLM，而是加入了证据评级：

```text
检索结果 → Grader 判断证据是否足够 → 不足时可触发联网补偿
```

这个设计体现了 Corrective RAG 的核心思想：

> 先判断证据质量，再决定是否补救，而不是盲目生成。

### 4.4 报告生成与引用校验

报告生成阶段已经支持：

- 引用编号
- 文件名
- 页码
- 来源标签
- 联网搜索结果整合
- 数据分析结果整合
- 引用校验

修复后，报告不再只说“文献片段不足”，而是能明确说明：

```text
CRAG_paper.pdf，第 1 页 / 第 2 页 / 第 9 页等
```

这让报告更像“可追溯的研究总结”，而不是普通聊天回答。

### 4.5 代码沙箱与安全控制

项目中包含自动生成分析代码的能力，因此做了安全限制：

- AST import 白名单过滤
- 禁止危险 import
- 子进程环境变量白名单
- 输出目录限制
- 沙箱执行错误解析

这个部分适合在面试中强调，因为它说明你不是只关注功能，也关注实际部署风险。

### 4.6 Benchmark 与验证

已完成：

- 单元测试：30/30 通过
- 冒烟测试：真实 CRAG 论文 PDF 跑通
- benchmark 页面从假控件变为真实可运行链路
- 测试集改为围绕真实 CRAG 论文的问题

验证结果：

```text
CRAG 论文解析：85 个 chunk
检索：可返回相关证据
报告：可生成带引用、带定量数据的 Markdown 研究报告
单元测试：30/30 通过
```

## 5. 开发过程中解决的关键问题

### 5.1 Chunk 缺少文档上下文

**问题**：

文档被切成 chunks 后，LLM 只看到孤立片段，报告中出现“缺少完整文章元数据”。

**解决**：

给 chunk / evidence 补充：

```text
filename
page_num
doc_id
source_label
```

**结果**：

报告可以明确引用文档来源和页码。

---

### 5.2 PDF 表格读不到

**问题**：

原 `chunk_pdf()` 只调用 `page.get_text()`，论文表格没有进入 chunk。

**解决**：

使用 PyMuPDF 的 `page.find_tables()` 提取表格，再转成文本追加到 chunk 中。

**结果**：

报告中可以出现具体定量数据，例如：

```text
PopQA 准确率 59.8%
FLOPs 27.2
执行时间 0.512s
```

---

### 5.3 沙箱继承环境变量风险

**问题**：

分析代码子进程如果继承完整环境变量，可能泄露 API Key 或本地敏感配置。

**解决**：

改为只传必要环境变量，例如：

```text
CRAG_OUTPUT_DIR
CRAG_WORK_DIR
PYTHONIOENCODING
```

**结果**：

安全边界更明确，适合未来云服务器部署。

---

### 5.4 过重模型依赖

**问题**：

BGE Reranker v2-m3 需要下载约 2.27GB 的 `model.safetensors`，对轻量项目不友好。

**解决**：

关闭 rerank，保留：

```text
向量检索 + BM25 + RRF 融合
```

**结果**：

系统更轻量，更容易在普通电脑或云服务器部署。

---

### 5.5 Benchmark 页面是假控件

**问题**：

原页面按钮不可用，无法真实运行 benchmark。

**解决**：

接通：

```text
TestSuite
BenchmarkRunner
Scorer
Comparator
```

**结果**：

页面可以真实加载测试集、运行基线、对比模型表现。

建议按这个顺序讲：

### 8.1 项目目标

> 我做的不是普通文档问答，而是一个轻量化 Adaptive CRAG 系统。目标是让系统能读长文档、检索证据、判断证据是否足够，并生成带引用的研究报告。

## 9. 面试可能追问与回答

### Q1：为什么不用普通 RAG？

普通 RAG 只负责检索和生成，但不会判断证据是否足够。这个项目用了 Corrective RAG 思路：先检索，再由 Grader 判断证据质量，不足时再补偿，比如联网搜索或进一步分析。

### Q2：为什么不用大 reranker？

项目目标是轻量化和可部署。重型 reranker 需要下载几 GB 模型，不适合普通云服务器和实习项目展示。因此保留向量检索、BM25 和 RRF 融合，保证效果与可运行性之间的平衡。

### Q3：PDF 表格怎么处理？

原来只用 `page.get_text()`，表格没有进入文本。后来使用 PyMuPDF 的 `page.find_tables()` 检测表格，再把表格转成可读文本追加进 chunk，使检索和报告阶段可以使用表格里的定量数据。

### Q4：自动生成代码为什么要沙箱？

因为 LLM 生成的代码不可信，不能直接在主进程执行。项目中通过 import 白名单、环境变量隔离和输出目录限制降低风险，避免代码访问敏感配置或执行危险操作。

### Q5：怎么验证项目真的跑通？

使用真实 CRAG 论文 PDF 做端到端验证：解析出 85 个 chunk，检索返回证据，报告能生成带页码引用和定量数据的 Markdown 研究报告；同时单元测试 30/30 通过。
